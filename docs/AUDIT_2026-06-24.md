## P0 — Field Supervisor (and dispatcher/sales/marketing) can open financial + growth screens by direct URL
**This is the exact thing the owner said must NOT happen** ("FS not seeing financials like accounting or growth intel").

The role **data model is correct** — `lib/roles.js:91` sets FS `seeFinancials:false, seeRevenue:false`, and the nav hides the Accounting + Sales/Marketing menus from FS. **But the page guards are the real gate, and they leak:**

| Page | Guard (today) | Leaks to |
|---|---|---|
| `app/(main)/invoices/page.js:10` | `requirePerm('seeFinancials','seeReports')` | fs, dispatcher, sales, marketing |
| `app/(main)/payments/page.js:11` | `requirePerm('seeFinancials','seeReports')` | fs, dispatcher, sales, marketing |
| `app/(main)/receipts/page.js:9` | `requirePerm('seeFinancials','seeReports')` | fs, dispatcher, sales, marketing |
| `app/(main)/revenue/page.js:22` | `requirePerm('seeFinancials','seeRevenue','seeReports')` | fs, dispatcher, sales, marketing |
| `app/(main)/cancel-insights/page.js:12` | `requirePerm('seeReports','seeFinancials','contactCustomer')` | fs |
| `app/(main)/top-customers/page.js:11` | `requirePerm('seeReports','seeRevenue','seeFinancials','seeAllJobs')` | fs |

`requirePerm` is **OR** (`canAny`, `lib/guard.js:32`). FS has `seeReports:true`, so the broad report perm alone opens the door — the nav only hides the link, it doesn't block the route.

**Fix (small):** the financial/intel screens must require the financial perm *specifically*, not OR it with `seeReports`:
- invoices/payments/receipts → `requirePerm('seeFinancials')`
- revenue → `requirePerm('seeRevenue','seeFinancials')`
- cancel-insights/top-customers → `requireRole([...])` or `requireHref(...)` with an explicit list that excludes fs.
Cleanest: switch these to `requireHref('/invoices')` etc. so the page reuses the nav gating that already excludes FS.

`payroll`, `bank-position`, `doc-fraud`, `cash-custody` are **not** affected — they use `requireRole([...])` and already exclude fs. ✅

## P1 — Booking confirmation defaults ON with consent pre-checked → a customer send can ride a bare "Book job" click
`BookingForm.js:58,60` — `serviceConsent` and `sendConfirm` both default `true`. A CSR who books and never touches the "Stay in touch" card sends the customer an SMS + email (`actions.js` confirm block). Server consent-gating is correct, but a **pre-ticked** consent box isn't an affirmative opt-in — it's the "didn't mean to send that" failure the no-auto-send rule exists to prevent (and pre-checked consent is weak TCPA posture).
**Fix:** default `serviceConsent=false` and `sendConfirm=false` so the CSR must consciously tick consent (after reading the on-screen script) and then tick send.

## P2 — worth fixing soon
- **dispatch.me tech text has no toggle + dead-link fallback.** `actions.js` warranty block fires on *every* warranty booking (re-sends on re-submit/edit), and if `DISPATCHME_APP_URL` is unset it texts a useless `https://dispatch.me` homepage link. Recipient is an internal tech (OK, and it's logged), but: gate it behind a "text the tech the dispatch.me link" checkbox, **skip the send when `DISPATCHME_APP_URL` is unset** (don't send a dead link), and dedupe on re-submit.
- **`is_plate` safety gate is not strict.** `actions.js:151` `if (!d.is_plate)` — a model returning the string `"false"` is truthy and **bypasses** the non-plate rejection. Change to `if (d.is_plate !== true)`.
- **Confirmation phone falls back to `cust.phones` (an array).** `actions.js` `phone || cust.phone || cust.phones` — if `phones` is a jsonb/array, `toE164(String(array))` can yield a wrong number or no-op. Use a scalar only (`cust.phone`, or `cust.phones?.[0]`).
- **Booking-confirmation email HTML isn't escaped.** `actions.js` inline template drops `nm`/`jobType`/`address` in raw; `lib/email.js` has `esc()`/`renderEmailHtml()` right there. Route through them.
